# v3.0 前半 設計メモ

## 目的

いちよんロボとイルシアを、同じソース・同じDB基盤で別Botインスタンスとして同時稼働できる形へ進める。
差分はソース分岐ではなく、`bot_id`、`guild_id`、DB設定、`.env`、Bot Token、機能ON/OFFで表現する。

このメモは現状調査に基づく実装方針であり、DB変更はまだ行わない。

## 現状

- 起動設定は `bot/config.py` が `.env` を読む。
- Bot実行は `main.py` に集約され、DB backend時は1分ループでモード期限切れと自動投稿を処理する。
- 管理画面は `/servers` から `guild_id` 単位で機能を管理する。
- 権限は `admin_users` と `guild_permissions` で、Botインスタンス単位の権限はまだない。
- 主要テーブルはほぼ `guild_id` 単位で、`bot_id` は未導入。
- デッキ検索は `mention_reactions.config_json` に設定を持ち、`bot/services/deck_search.py` と `bot/services/x_search.py` が実行する。
- 特殊効果の `probability_multiplier` は `runtime_db.py` の共通重み・確率処理で参照される。
- リアクション閾値返信は `reaction_threshold_rules.config_json` と `reaction_threshold_events` で管理する。

## 追加したい共通軸

### BOT_INSTANCE_ID

起動時に `BOT_INSTANCE_ID` を読む。

候補:

- `ichiyon`
- `irsia`

未設定時は後方互換として `ichiyon` 扱いにする。

### bot_instances

将来の管理画面表示と権限制御のため、次のようなテーブルを検討する。

```sql
CREATE TABLE bot_instances (
    bot_id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    enabled BOOLEAN NOT NULL DEFAULT TRUE,
    description TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

TokenはDBへ保存しない。TokenはComposeまたはsystemd側の環境変数で分ける。

## bot_id を持たせる候補

既存の `guild_id` だけでは、同じDiscordサーバーに複数Botを入れた時に設定が混ざる。
段階移行では、まずnullableまたはdefault付きで `bot_id='ichiyon'` を足し、Repository側で `BOT_INSTANCE_ID` を条件に加える。

対象候補:

- `feature_flags`
- `mention_reactions`
- `mention_reaction_choices`
- `mention_search_handlers`
- `mention_limited_effects`
- `reactions`
- `ng_words`
- `special_effect_tags`
- `special_effect_assignments`
- `counters`
- `counter_states`
- `modes`
- `mode_reply_choices`
- `mode_trigger_conditions`
- `mode_exit_conditions`
- `mode_states`
- `mode_trigger_history`
- `auto_posts`
- `auto_post_delivery_history`
- `reaction_threshold_rules`
- `reaction_threshold_events`
- `deck_settings` または後続のデッキ検索設定テーブル

`guilds` はDiscordサーバー自体なので、原則 `bot_id` を持たせなくてもよい。
ただしBotごとにサーバー表示名や有効状態を分けたい場合は `bot_guilds(bot_id, guild_id, enabled, display_name)` を追加する。

## X更新通知

### 方針

既存のX Bearer Token、httpx、ログ方針は `bot/services/x_search.py` から流用する。
ただしデッキ検索とは用途が違うため、サービスは分ける。

候補:

- `bot/services/x_notifications.py`
- `bot/repositories/x_notifications.py`

### DB案

```sql
CREATE TABLE x_watch_accounts (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    bot_id TEXT NOT NULL DEFAULT 'ichiyon',
    guild_id TEXT NOT NULL REFERENCES guilds(guild_id) ON DELETE CASCADE,
    x_user_id TEXT NOT NULL,
    x_username TEXT NOT NULL,
    enabled BOOLEAN NOT NULL DEFAULT TRUE,
    include_replies BOOLEAN NOT NULL DEFAULT FALSE,
    include_retweets BOOLEAN NOT NULL DEFAULT FALSE,
    include_quotes BOOLEAN NOT NULL DEFAULT FALSE,
    interval_minutes INTEGER NOT NULL DEFAULT 15,
    last_seen_post_id TEXT,
    last_posted_post_id TEXT,
    last_checked_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (bot_id, guild_id, x_user_id)
);

CREATE TABLE x_watch_targets (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    bot_id TEXT NOT NULL DEFAULT 'ichiyon',
    guild_id TEXT NOT NULL REFERENCES guilds(guild_id) ON DELETE CASCADE,
    watch_account_id BIGINT NOT NULL REFERENCES x_watch_accounts(id) ON DELETE CASCADE,
    channel_id TEXT NOT NULL,
    enabled BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (bot_id, guild_id, watch_account_id, channel_id)
);

CREATE TABLE x_post_delivery_history (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    bot_id TEXT NOT NULL DEFAULT 'ichiyon',
    guild_id TEXT NOT NULL REFERENCES guilds(guild_id) ON DELETE CASCADE,
    watch_account_id BIGINT NOT NULL REFERENCES x_watch_accounts(id) ON DELETE CASCADE,
    post_id TEXT NOT NULL,
    channel_id TEXT NOT NULL,
    delivered_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (bot_id, guild_id, watch_account_id, post_id, channel_id)
);
```

### 初回登録

- X APIで対象アカウントの最新Post IDだけ確認する。
- `last_seen_post_id` に保存する。
- 過去投稿は流さない。
- 通常投稿だけ初期ON。返信、リポスト、引用は初期OFF。

### 通常監視

- 1分ループ内で、`last_checked_at + interval_minutes` を過ぎた購読だけ処理する。
- 前回以降の新着だけ取得する。
- 取得できるAPI方式は実装時に公式仕様で確認する。
- 投稿済み判定は `last_posted_post_id` と `x_post_delivery_history` の両方で守る。
- API失敗時はBot全体を止めず、短いログだけ残す。

## デッキ検索の取得開始日

### 現状

デッキ検索は `mention_reactions` の `reaction_key='deck_search'` と `config_json` で設定する。
現在は `lookback_days` と `search_mode` で検索範囲を決める。

### 追加設定案

bot/guild単位にするなら、将来的には次のどちらか。

1. `mention_reactions.config_json.fetch_since_date`
2. `deck_search_settings(bot_id, guild_id, fetch_since_date, updated_by, updated_at)`

v3.0で `bot_id` を入れるなら、2の専用テーブルが安全。
ただし短期実装では既存 `config_json` に入れる方が小さい。

### Discordコマンド案

- `デッキ 取得日更新 6/27から`
- `デッキ 取得日確認`
- `デッキ 取得日リセット`

処理場所:

- `bot/services/runtime_db.py` のメンション処理で、デッキ検索へ入る前に管理コマンドとして判定する。
- 権限は開発者またはDB管理権限を確認する。

仕様:

- 年なし日付は現在年で補完する。
- `fetch_since_date` がある時はX検索の `start_time` に使う。
- 未設定なら既存の `lookback_days` を使う。
- 古すぎる日付は拒否する。上限はX API契約や運用費に合わせて設定する。
- `高精度`、追加検索語、フォーマット指定とは独立させる。

## 管理画面改善

### コピー作成

対象:

- メンション反応
- メンション反応候補
- 自動反応
- NGワード
- 特殊効果タグ
- モード
- モード返答候補
- 自動投稿
- リアクション閾値ルール
- X通知ルール

共通仕様:

- コピー後は `enabled=false`。
- `id`、`created_at`、`updated_at`、実行履歴、`last_run_at`、`last_seen_post_id`、`last_posted_post_id` はコピーしない。
- 子レコードをコピーするかは機能ごとに明示する。

### 一括ON/OFF

一覧画面でチェックボックス選択、POSTで一括更新する。
Repositoryには `bulk_set_enabled(guild_id, ids, enabled)` を追加する。
`bot_id` 導入後は `bot_id` も条件に含める。

### 横断表示

開発者またはglobal_adminだけ、Botごとの全設定を横断表示する。
一般管理者は `bot_permissions` と `guild_permissions` の交差範囲だけ表示する。

権限案:

```sql
CREATE TABLE bot_permissions (
    bot_id TEXT NOT NULL REFERENCES bot_instances(bot_id) ON DELETE CASCADE,
    discord_user_id TEXT NOT NULL,
    role TEXT NOT NULL CHECK (role IN ('bot_admin', 'viewer')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (bot_id, discord_user_id)
);
```

## 特殊効果の倍率上限

### 現状

- `probability_multiplier` は `effect_config_json.multiplier` を読む。
- `runtime_db.get_probability_multiplier_for_target()` で複数効果を乗算する。
- `choose_weighted_row_with_effects()` と `probability_hit_with_multiplier()` の両方で参照される。
- 直近のテストでは、同じlabelの重ね掛けを重複排除せず保持する確認がある。

### 上限案

設定場所候補:

1. 個別タグの `effect_config_json.max_multiplier`
2. guild/bot単位の `feature_settings`
3. 特殊効果全体の `special_effect_runtime_settings`

短期は1が小さい。
v3.0でbot別にするなら2か3が管理しやすい。

例:

```json
{
  "multiplier": 9,
  "label": "raio",
  "max_effective_multiplier": 729,
  "enabled": true
}
```

判定:

- 累積乗算後に `max_effective_multiplier` を超えたら丸める。
- 未設定なら現行通り上限なし。
- Bot/guild全体上限がある場合は、個別タグ上限より強い制限として扱う。

## Compose / .env

### ローカル

`docker-compose.yml` は `db`、`admin`、`bot` を持つ。
botはprofile指定。

### 本番

`docker-compose.prod.yml` は本番コンテナ名と公開ポートだけを上書きする。

### v3.0案

同じイメージから複数Botを起動する。

例:

```yaml
services:
  bot-ichiyon:
    profiles: ["bot"]
    environment:
      BOT_INSTANCE_ID: ichiyon
      DISCORD_TOKEN: ${DISCORD_TOKEN}

  bot-irsia:
    profiles: ["irsia"]
    environment:
      BOT_INSTANCE_ID: irsia
      DISCORD_TOKEN: ${IRSIA_DISCORD_TOKEN}
```

秘密値は `.env` にのみ置く。
`.env.example` には空値または `replace_me` のみ書く。

## 実装順序

1. `BOT_INSTANCE_ID` を読むだけの準備を入れる。
   - 未設定時は `ichiyon`。
   - ログに `BOT_INSTANCE_ID` を出す。
   - まだDB検索条件には使わない。
2. `bot_instances` と `bot_permissions` を追加する。
   - 既存環境は `ichiyon` をseedする。
3. 主要テーブルへ `bot_id` を後方互換で追加する。
   - nullableまたはdefault `ichiyon`。
   - 既存unique制約を `(bot_id, guild_id, ...)` に移行する。
4. Repositoryへ `bot_id` 条件を追加する。
   - 関数引数は増やしすぎず、Repository生成時に `bot_id` を持たせる案も検討する。
5. 管理画面にBot選択と権限制御を追加する。
6. デッキ検索の取得開始日設定を追加する。
7. X更新通知のDB、Repository、サービス、管理画面を追加する。
8. コピー作成、一括ON/OFF、横断表示を追加する。
9. 特殊効果倍率上限を追加する。

## 次回以降の注意

- 既存本番は `main` + Docker Composeで動いているため、v3.0のDB migrationは必ずstgでdump/restore検証する。
- `bot_id` 追加は広範囲なので、一度に全機能を切り替えない。
- まず `bot_id` 未指定なら現行挙動、指定時だけ絞り込む形にする。
- X通知は費用対策として初回の過去投稿流し込みを禁止する。
- X通知の実装時は、使うX APIエンドポイントと差分取得パラメータを公式仕様で再確認する。
