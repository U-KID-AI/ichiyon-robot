# いちよんロボ ver2.0 ロードマップ

## 目的

ver2.0 は、いちよんロボを複数サーバーで安全に運用できる管理Botへ進化させる大型改修です。
単にネタ機能を増やすのではなく、サーバー別管理、Discordログイン管理画面、機能ON/OFF、DB化、デッキ検索基盤を整えます。

```txt
いちよんロボ ver2.0
= 複数サーバー対応 + Discordログイン管理画面 + サーバー別機能ON/OFF + DB化 + デッキ検索基盤
```

## 現在の前提

ver1.0 はリリース済みで、現在の `main` は安定版です。
直近では自動反応に優先度を追加済みです。

ver1.0 の主な既存機能は以下です。

* 名言ランダム返答
* おみくじ
* 自動反応
* 自動反応の優先度
* NGワード
* 画像付き返答
* はゆすモード
* 6/30自動投稿
* FastAPI管理画面
* 本番/開発Bot分離
* Oracle VMで常時起動

ver2.0 では、既存機能を壊さずに管理基盤を作り直します。

## 対象サーバー

### いちよんラボ

開発・検証用サーバーです。
危険機能や新機能のテストを行うため、基本的には全機能ON寄りで運用します。

### ランセ地方

メイン運用サーバーです。
名言、おみくじ、自動反応、NGワード、はゆす、自動投稿などを使います。
暴れる機能は制限つきにします。

### レストラン

シャドバ用サーバーです。
デッキ検索機能を主役にします。
はゆす、成田、破壊、しこっちは基本OFF想定です。

## 実装フェーズ

### Phase 0: 設計ドキュメント追加

今回の作業です。
実装前に、管理画面、DB、機能別方針、モード設計を文書化します。

この段階では既存Pythonコード、Bot挙動、Docker構成、OAuth、管理画面実装は変更しません。

### Phase 1: PostgreSQL 基盤の検証

PostgreSQL を第一候補として、ローカル開発で接続できる状態を作ります。

推奨順は以下です。

1. PostgresだけDockerで起動する
2. ローカルPythonからPostgresへ接続する
3. 既存JSONデータから移行する方針を決める
4. 最小限のテーブルで読み書きを検証する

### Phase 2: データモデル移行

すべての運用データを基本的に `guild_id` でサーバー別に分離します。
既存機能の振る舞いを保ったまま、DB上のモデルへ段階的に寄せます。

対象データは以下です。

* quotes
* kuji_results
* reactions
* ng_words
* feature_flags
* modes
* mode_states
* auto_posts
* special_effect_tags
* deck_settings
* admin_users / guild_admins

### Phase 3: Discordログイン管理画面

Discord OAuth2 でログインし、DiscordユーザーIDで権限判定します。
独自のID/パスワードは持ちません。

画面遷移は以下です。

```txt
Discordログイン
↓
サーバー選択
↓
機能一覧
↓
各機能編集画面
```

### Phase 4: サーバー別機能ON/OFF

サーバーごとに機能一覧を表示し、機能単位で有効/無効を切り替えます。
機能一覧では ON/OFF スイッチと編集画面への遷移を分けます。

候補機能は以下です。

* 名言
* おみくじ
* 自動反応
* NGワード
* モード
* 自動投稿
* デッキ検索
* 特殊効果タグ
* 破壊

### Phase 5: デッキ検索基盤

デッキ検索は ver2.0 の目玉機能の一つです。
Botにデッキを保存する機能ではなく、Xから条件に合う最新ポストを検索して貼る機能として設計します。

コマンド例:

```txt
@いちよんロボ デッキ エルフ
@いちよんロボ デッキ エルフ ローテーション
```

処理イメージ:

```txt
Xから該当ポスト検索
↓
BEYOND到達っぽい投稿を候補にする
↓
添付画像にQRコードがあるか確認
↓
画像内のクラスアイコンが指定クラスか判定
↓
条件に合うポストを2件程度返す
```

### Phase 6: Docker化

最終的には以下をまとめて起動できるようにします。

* Bot
* 管理画面
* PostgreSQL

段階的に進め、最初から全サービスをDocker化しない方針です。

## 権限設計

Discord OAuth2ログイン後、DiscordユーザーIDで権限判定します。

権限候補:

* `global_admin`: 全サーバー管理可能
* `guild_admin`: 自サーバー管理可能
* `editor`: 許可されたサーバーの通常データ編集可能
* `viewer`: 閲覧のみ

危険設定は管理者限定にします。

危険設定例:

* 機能ON/OFF
* モード有効化
* 特殊効果タグ作成/編集
* 破壊設定
* 権限設定

## 既存運用上の注意

Python 3.8対応を維持します。
`str | None` や `list[str]` など、Python 3.10前提の型ヒントは使いません。

秘密情報はコミットしません。

コミット禁止:

* `.env`
* Bot Token
* 秘密鍵
* `data/backups/`
* 本番データ
* 画像アップロード実体
